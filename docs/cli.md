# noxuslab CLI

Installed by `pip install noxuslab` (or `pip install -e .[dev]` from a
clone). Also runnable as `python -m noxuslab`.

## Synopsis

    noxuslab init <dir>          [--interactive | --no-interactive]
    noxuslab doctor
    noxuslab pull <workflow_id>  [-o PATH | -o -] [-f]
    noxuslab push <file>         [--dry-run]
    noxuslab diff <workflow_id> <file> [--visual]
    noxuslab run  <id-or-file>   [--input k=v]... [--detach]
    noxuslab trace [list | show <id>] [--limit N] [--json]
    noxuslab env   [list | use <name>]
    noxuslab list
    noxuslab show  <workflow_id>
    noxuslab agents [list | pull | push | diff | delete] ...
    noxuslab chat   [-a <agent>] [-m <model>]
    noxuslab ask    "<question>" [-a <agent>] [-m <model>]
    noxuslab gen    "<prompt>"   [-a <agent>] [-m <model>] [-o PATH]
    noxuslab watch  <file>       [--interval SECS]
    noxuslab fmt    <file>...    [--check] [--diff]
    noxuslab portal              [--host HOST] [--port PORT] [--no-open]
    noxuslab mcp serve           [--transport stdio|sse]
    noxuslab version
    noxuslab --version | -V

All commands except `init`, `doctor`, and `version` need
`NOXUS_API_KEY` in your environment (loaded from `.env`
automatically). Optional: `NOXUS_BACKEND_URL` for self-hosted
instances. See [../SECURITY.md](../SECURITY.md) for
`NOXUSLAB_SECRETS_CMD` and `NOXUSLAB_AUDIT_LOG`.

## init

    noxuslab init my-noxus-project
    noxuslab init my-noxus-project --no-interactive

Scaffolds a fresh project: copies `examples/` and `.env.example`,
writes a minimal `README.md`, and pins
`.noxuslab-template-version`. Refuses to scaffold into a non-empty
directory. The interactive wizard prompts for `NOXUS_API_KEY`
(hidden) and an optional backend URL, then writes a chmod-600
`.env`. Skip with Ctrl+C — the scaffold is already on disk.

The CLI lives in your virtualenv, not in the project tree. Upgrade
later with:

    pip install --upgrade git+https://github.com/AdvanceWorks/noxus-lab.git

## doctor

    noxuslab doctor

Diagnoses the project: Python version, virtualenv, `noxus-sdk`
import, `.env` presence, `NOXUS_API_KEY` resolution, and a single
network probe to the Noxus backend. Exits 0 when everything is
green, non-zero on the first hard failure.

## pull

    noxuslab pull <id>
    noxuslab pull <id> -o my_flow.py
    noxuslab pull <id> -o -            # to stdout
    noxuslab pull <id> --force         # overwrite existing file

Without `-o`, writes `examples/NN_<slug>.py` (next free `NN`). The
generated file is self-contained and carries a provenance header
(`# generated by noxuslab pull from <id> @ <utc>`).
See [codegen.md](codegen.md) for rendering details.

## push

    noxuslab push examples/NN_my_flow.py
    noxuslab push examples/NN_my_flow.py --dry-run

Imports the file via `runpy.run_path` (sandboxed namespace, no
`exec`), expects a top-level `wf: WorkflowDefinition`, then calls
`client.workflows.save(wf)` and prints the resulting id. The
trailing `print(c.workflows.save(wf).id)` line in pulled files is
stripped before exec to avoid double-saving. `--dry-run` validates
structure and skips the save.

> Security: only push code you wrote or reviewed.

## diff

    noxuslab diff <id> examples/NN_my_flow.py
    noxuslab diff <id> examples/NN_my_flow.py --visual

Pulls the current server state, generates the canonical Python form,
and unified-diffs it against your local file. Exits **0** if
identical, **1** if there are differences (chainable:
`noxuslab diff <id> <file> && noxuslab push <file>`).

`--visual` emits two Mermaid `flowchart TD` blocks (server vs local)
instead of text — paste into mermaid.live or VS Code's preview.

## run

    noxuslab run <workflow-id> --input topic="octopus cognition"
    noxuslab run examples/NN_my_flow.py --input k=v --detach

Triggers a workflow run with the given inputs, follows it until
completion (or `--detach`s), and prints the final output. Accepts
either a workflow id or a local Python file; for files the workflow
is pushed first, then run. Each run is recorded under
`.noxuslab/traces/<run-id>/` for `noxuslab trace show`.

## trace

    noxuslab trace                          # = trace list
    noxuslab trace list --limit 50
    noxuslab trace show <run-id>
    noxuslab trace show <run-id> --json

`list` shows the most recent local runs (one row each). `show`
prints the captured inputs, per-node timings, output, and any error.
`--json` emits a machine-readable form for piping.

## env

    noxuslab env                            # = env list
    noxuslab env list
    noxuslab env use staging

Switches between named API-key environments stored under
`~/.config/noxuslab/envs/`. Each env is a tiny dotenv file with
`NOXUS_API_KEY` (and optional `NOXUS_BACKEND_URL`). The active env
is symlinked into the current project's `.env`.

## list, show

    noxuslab list                           # id  name
    noxuslab show <workflow_id>             # full wire JSON

## agents

    noxuslab agents                          # list (id  name)
    noxuslab agents list                     # explicit form
    noxuslab agents pull <id>                # writes agents/NN_<slug>.py
    noxuslab agents pull <id> -o my_a.py     # explicit out path
    noxuslab agents pull <id> -o -           # to stdout
    noxuslab agents push my_a.py             # create or update
    noxuslab agents push my_a.py --dry-run   # validate, don't save
    noxuslab agents diff <id> my_a.py        # exit 1 if differs
    noxuslab agents delete <id> --yes        # destructive, requires --yes

The pulled file is a single self-contained script with three
top-level variables: `agent_name` (str), `agent_id` (str — remove
or set empty to force creating a NEW agent on push), and
`agent_settings` (`ConversationSettings`). Push runs the file with
the SDK `Client` stubbed (no network), reads those three variables,
then calls `client.agents.update(...)` if `agent_id` is set or
`client.agents.create(...)` otherwise.

## chat, ask

    noxuslab chat                            # default model: gemini-2.5-flash
    noxuslab chat -a <agent_id>              # tied to an agent (uses its tools/KB)
    noxuslab chat -m gpt-4o                  # pick a different model
    noxuslab ask  "what's in our Q3 KB?" -a <agent_id>
    echo "summarise the docs" | noxuslab ask -a <agent_id>

`chat` is an interactive REPL with SSE-streamed responses; type
`/exit` to quit, `/clear` to start a new conversation. `ask` is the
one-shot, pipe-friendly variant — perfect for shell scripts.

## gen

    noxuslab gen "summarise PDFs and email them weekly"
    noxuslab gen "build an FAQ bot" --agent <workflow-aware-agent-id>
    noxuslab gen "..." --out examples/99_my_idea.py

Asks a Noxus agent for Python code that defines a
`WorkflowDefinition` named `wf`. Strips markdown fences. Writes to
`examples/NN_<slug>.py` (or `--out`). Review the file before
`noxuslab push`. Free-form prompts are never written to the audit
log.

## watch

    noxuslab watch examples/05_my_flow.py
    noxuslab watch examples/05_my_flow.py --interval 0.2

Polls the file's mtime; on every save, executes the file and pushes
the resulting `wf` to Noxus. First push fires immediately. Each
push prints a timestamp + new workflow id + elapsed ms. Hot-reload
for workflows. Ctrl+C to stop.

## fmt

    noxuslab fmt examples/05_my_flow.py
    noxuslab fmt examples/*.py --check       # exit 1 if any need reformat
    noxuslab fmt examples/05_my_flow.py --diff  # preview without writing

The workflow analogue of `ruff format`. Loads each file (the SDK
`Client` is stubbed during the load — no network), regenerates the
canonical Python via the same renderer used by `pull`, and rewrites
the file in place. Provenance headers are preserved.

- exit `0` — file already canonical (or successfully reformatted)
- exit `1` — `--check` mode and changes are pending, or `--diff`
  mode printed a non-empty diff

## portal

    noxuslab portal                          # http://127.0.0.1:7890
    noxuslab portal --port 8080
    noxuslab portal --no-open                # do not auto-open browser

Read-only HTML dashboard. Lists workflows and agents in the
connected workspace, with one-click links back to the Noxus UI.
Useful for a quick visual scan without leaving the terminal.

## mcp serve

    noxuslab mcp serve                       # stdio (default; for editors)
    noxuslab mcp serve --transport sse

Exposes the noxuslab CLI as a Model Context Protocol server so
editors and other agents can call `pull`, `push`, `list`, etc. via
MCP tools. Requires the `mcp` extra: `pip install 'noxuslab[mcp]'`.
See [mcp.md](mcp.md).

## version

    noxuslab version
    noxuslab -V

Prints the installed `noxuslab` version. Handy in bug reports.
