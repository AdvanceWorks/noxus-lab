# noxuslab CLI

Tiny CLI installed by `make setup` (via `pip install -e .[dev]`).
Also runnable as `python -m noxuslab`.

## Synopsis

    noxuslab pull <workflow_id> [-o PATH | -o -] [-f]
    noxuslab push <file>        [--dry-run]
    noxuslab diff <workflow_id> <file> [--visual]
    noxuslab watch <file>       [--interval SECS]
    noxuslab gen "<prompt>"     [-a <agent>] [-m <model>] [-o PATH]
    noxuslab fmt <file>...      [--check] [--diff]
    noxuslab portal             [--host HOST] [--port PORT] [--no-open]
    noxuslab run <target>       [--input k=v]... [--detach]
    noxuslab trace [list|show <id>] [--limit N] [--json]
    noxuslab env  [list|use <name>]
    noxuslab doctor
    noxuslab list
    noxuslab agents
    noxuslab show <workflow_id>
    noxuslab chat [-a <agent_id>] [-m <model>]
    noxuslab ask  <question> [-a <agent_id>] [-m <model>]
    noxuslab init <dir> [--with-makefile] [--interactive | --no-interactive]
    noxuslab mcp serve [--transport stdio|sse]
    noxuslab version
    noxuslab --version | -V

All commands except `init`/`version` need `NOXUS_API_KEY` in your
environment (loaded from `.env` automatically). Optional:
`NOXUS_BACKEND_URL` for self-hosted instances. See
[security.md](security.md) for `NOXUSLAB_SECRETS_CMD` and
[for-builders.md](for-builders.md) for production env vars.

## pull

    noxuslab pull 1b115224-aaaa-bbbb-cccc-...
    noxuslab pull <id> -o my_flow.py
    noxuslab pull <id> -o -            # to stdout (pipe-friendly)
    noxuslab pull <id> --force         # overwrite existing file

Without `-o`, writes `examples/NN_<slug>.py` (next free `NN`). The
generated file carries a provenance header (`# generated by noxuslab
pull from <id> @ <utc>`) and is self-contained: run it to create a
*new* workflow with the same structure.

See [codegen.md](codegen.md) for rendering details.

## push

    noxuslab push examples/NN_my_flow.py
    noxuslab push examples/NN_my_flow.py --dry-run

Imports the file via `runpy.run_path` (sandboxed namespace, no
`exec`), expects a top-level `wf: WorkflowDefinition`, then calls
`client.workflows.save(wf)` and prints the resulting id. The trailing
`print(c.workflows.save(wf).id)` line in pulled files is stripped
before exec to avoid double-saving. `--dry-run` validates structure
and skips the save.

> Security: only push code you wrote or reviewed.

## list, show

    noxuslab list                      # id  name
    noxuslab show <workflow_id>        # full wire JSON

## diff

    noxuslab diff <workflow_id> examples/NN_my_flow.py

Pulls the current server state, generates the canonical Python form,
and unified-diffs it against your local file. Exits **0** if identical,
**1** if there are differences (so you can chain it in a script:
`noxuslab diff <id> <file> && noxuslab push <file>`).

## agents

    noxuslab agents                          # list (id  name)
    noxuslab agents list                     # explicit form
    noxuslab agents pull <id>                # writes agents/NN_<slug>.py
    noxuslab agents pull <id> -o my_a.py     # explicit out path
    noxuslab agents pull <id> -o -           # to stdout
    noxuslab agents push my_a.py             # create or update
    noxuslab agents push my_a.py --dry-run   # validate, don't save
    noxuslab agents diff <id> my_a.py        # exit 1 if differs
    noxuslab agents delete <id> --yes        # destructive, requires --yes

The pulled file is a single self-contained script with three top-level
variables: `agent_name` (str), `agent_id` (str — remove or set empty to
force creating a NEW agent on push), and `agent_settings`
(`ConversationSettings`). Push runs the file with the SDK `Client`
stubbed (no network), reads those three variables, then calls
`client.agents.update(...)` if `agent_id` is set or
`client.agents.create(...)` otherwise.

## chat, ask

    noxuslab chat                      # default model: gemini-2.5-flash
    noxuslab chat -a <agent_id>        # tied to an agent (uses its tools/KB)
    noxuslab chat -m gpt-4o            # pick a different model
    noxuslab ask "what's in our Q3 KB?" -a <agent_id>
    echo "summarise the docs" | noxuslab ask -a <agent_id>

`chat` is an interactive REPL with SSE-streamed responses; type
`/exit` to quit, `/clear` to start a new conversation. `ask` is the
one-shot, pipe-friendly variant — perfect for shell scripts.

## init

    noxuslab init my-noxus-project
    noxuslab init my-noxus-project --with-makefile

Scaffolds a fresh project: copies `examples/` and `.env.example` and
writes a minimal `README.md`. With `--with-makefile`, also copies the
top-level `Makefile` and `bin/` so the new project has the same `make
help` menu as this repo. Refuses to scaffold into a non-empty
directory. Writes `.noxuslab-template-version` so `make
template-update` can later show what changed upstream.

This is the default way to start a new Noxus project. The GitHub repo
itself is primarily the source for the CLI and examples; most users
should `pip install ...` and run `noxuslab init ...` instead of cloning
the full repo.

The generated project keeps the CLI in your virtualenv, not in the
project tree. Upgrading later is a reinstall:

    pip install --upgrade git+https://github.com/AdvanceWorks/noxus-lab.git

The minimal scaffold does **not** assume `make`. Only `--with-makefile`
adds `Makefile` + `bin/` wrappers.

## version

    noxuslab version
    noxuslab -V

Prints the installed `noxuslab` version. Handy in bug reports.

## watch

    noxuslab watch examples/05_my_flow.py
    noxuslab watch examples/05_my_flow.py --interval 0.2

Polls the file's mtime; on every save, executes the file and pushes the
resulting `wf` to Noxus. First push fires immediately. Each push prints
a timestamp + new workflow id + elapsed ms. Hot-reload for workflows.
Ctrl+C to stop.

## gen

    noxuslab gen "summarise PDFs and email them weekly"
    noxuslab gen "build an FAQ bot" --agent <workflow-aware-agent-id>
    noxuslab gen "..." --out examples/99_my_idea.py

Asks a Noxus agent for Python code that defines a `WorkflowDefinition`
named `wf`. Strips markdown fences. Writes to `examples/NN_<slug>.py`
(or `--out`). Review the file before `noxuslab push`. Free-form prompts
are never written to the audit log.

## diff --visual

    noxuslab diff <workflow_id> <file> --visual

Emits two Mermaid `flowchart TD` blocks (server vs local). Paste into
GitHub, mermaid.live, or VS Code's Mermaid preview to see the graphs
side by side. Without `--visual`, prints a unified Python diff.

## init wizard

    noxuslab init my-project              # interactive when stdin is a TTY
    noxuslab init my-project --no-interactive
    noxuslab init my-project --interactive

The interactive wizard prompts for `NOXUS_API_KEY` (hidden via
`getpass`) and an optional backend URL, then writes a chmod-600 `.env`
inside the new project. Skip with Ctrl+C — the scaffold is already on
disk.

## fmt

    noxuslab fmt examples/05_my_flow.py
    noxuslab fmt examples/*.py --check       # exit 1 if any need reformat
    noxuslab fmt examples/05_my_flow.py --diff  # preview without writing

The workflow analogue of `ruff format`. Loads each file (the SDK
`Client` is stubbed during the load — no network), regenerates the
canonical Python via the same renderer used by `pull`, and rewrites the
file in place. Provenance headers (`# generated by noxuslab pull from
<id>`) are preserved.

- exit `0` — file already canonical (or successfully reformatted)
- exit `1` — `--check` mode and changes are pending, or `--diff` mode
  printed a non-empty diff

## portal

    noxuslab portal                          # http://127.0.0.1:7890
    noxuslab portal --port 8080
    noxuslab portal --no-open                # do not auto-open browser

Read-only HTML dashboard. Lists workflows and agents in the connected
workspace and offers a one-click "copy id" button for each. Built on
the stdlib `http.server` — zero extra dependencies. Refuses to bind
on anything other than `127.0.0.1` to avoid accidental network
exposure. Ctrl+C in the terminal stops it.

## run

    noxuslab run examples/05_my_flow.py --input topic="space"
    noxuslab run 1b115224-aaaa-bbbb-cccc-... --input n=42 --input items='[1,2]'
    noxuslab run my_flow.py --input data=@payload.json   # load JSON from file
    noxuslab run <id_or_file> --detach                   # trigger and exit

Executes a workflow and streams events to stdout while recording a
JSONL trace under `.noxuslab/traces/<utc>_<run_id>.jsonl`. The
`<target>` is either a server-side workflow UUID (executed as-is) or a
local Python file (saved first via `client.workflows.save`, then
executed). `--input k=v` is repeatable; values are JSON-decoded when
possible (`true`, `42`, `[1,2]`), otherwise treated as plain strings.
Prefix a value with `@` to load it from a JSON file.

Ctrl+C detaches the client without cancelling the server-side run.

## trace

    noxuslab trace                       # list (alias for `trace list`)
    noxuslab trace list --limit 50
    noxuslab trace show <run_id>         # render as a timeline
    noxuslab trace show <prefix> --json  # raw JSONL entries

Each `run` writes a trace file. `trace list` shows recent runs newest
first with status + total elapsed. `trace show` renders a timeline:
header → events (relative timestamps, node names) → footer (status,
elapsed, output snippet). API keys, tokens, and other secret-shaped
keys are redacted before being written to disk.

## env

    noxuslab env                         # list (active marked with *)
    noxuslab env use dev                 # copy .env.dev → .env
    noxuslab env use prod                # copy .env.prod → .env

Switch between `.env.<name>` files for dev/staging/prod separation.
The chosen file is **copied** to `.env` (not symlinked, so it works on
Windows without admin and edits to `.env` don't leak back to the
source). The active name is recorded in `.noxuslab/active-env`.

Add the per-env files to `.gitignore` if they contain secrets. A
sensible layout:

    .env.dev      # local dev key, points to staging backend
    .env.staging  # CI key
    .env.prod     # production key (CI-managed)
    .env          # generated by `noxuslab env use ...`

## doctor

    noxuslab doctor

Run sanity checks on the local setup. Reports:

- Python version
- `noxuslab` version
- `noxus_sdk` import works
- `NOXUS_API_KEY` resolvable (env / `.env` / `NOXUSLAB_SECRETS_CMD`)
- backend reachable (TCP connect, 2s timeout — warn only)
- `.noxuslab/traces/` writable
- active env (informational)

Exit code is `0` when all hard checks pass, `1` otherwise.
